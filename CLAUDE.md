# Plan Mode 규칙

- 계획 작성은 구조적으로, 하지만 각 항목은 간결하게
- 불필요한 세부사항은 생략

## 파일명

- plan 파일명은 내용과 관련된 설명적 이름으로 지정할 것
- 이름 형식: `<동작>-<대상>-<맥락>.md` (예: `setup-auth-system.md`, `refactor-db-schema.md`)
- 랜덤 이름(예: `sparkling-napping-shannon.md`) 사용 금지

## 플랜 문서 구조

모든 플랜 문서는 다음 섹션을 포함할 것:

### 1. 배경 (Context)

- 왜 이 변경이 필요한지 (문제 또는 요구사항)
- 현재 어떻게 동작하는지 (as-is)
- 변경 후 어떻게 동작해야 하는지 (to-be)

### 2. 탐색 요약 (Codebase Summary)

- 관련 파일 목록과 각 파일의 역할 (경로 포함)
- 재사용할 기존 함수/유틸리티/패턴 (경로:라인 포함)
- 참고할 유사 구현 사례 (이미 프로젝트에 비슷한 패턴이 있으면 명시)

### 3. 구현 계획 (Implementation Steps)

- 번호가 매겨진 단계별 목록
- 각 단계마다: 대상 파일, 변경 내용, 검증 방법을 명시
- 형식:
    ```
    1. [파일경로] - 변경 내용 설명
       검증: 테스트 실행 / 빌드 확인 / 수동 확인 방법
    ```

### 4. 영향 범위 (Impact Analysis)

- 수정 대상 파일 수와 목록
- 이 변경으로 영향받는 다른 기능/컴포넌트
- 주의해야 할 사이드 이펙트

### 5. 검증 계획 (Verification)

- 구현 완료 후 전체 검증 방법 (테스트, 빌드, 수동 확인)

## 탐색 원칙

- 구현을 제안하기 전에 기존 코드의 패턴과 컨벤션을 충분히 파악할 것
- 유사한 기능이 이미 구현되어 있다면 해당 구현 방식을 따를 것
- 새로운 패턴을 도입하기 전에 기존 패턴으로 해결 가능한지 먼저 확인할 것
